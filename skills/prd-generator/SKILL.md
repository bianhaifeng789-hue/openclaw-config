# PRD 产品需求文档生成技能

当用户提到 **PRD、产品文档、需求文档、功能规格、MVP、版本规划、需求拆解** 时，使用本技能。

---

## 1. 适用场景
- 新产品立项
- 新功能设计
- 旧功能重构
- 竞品对标后方案落地
- MVP 范围界定
- 研发评审前需求整理

---

## 2. 输入
支持输入：
- 一句话产品想法
- 功能需求列表
- 竞品分析材料
- APK / Google Play 研究结论
- 业务目标 / 指标目标
- `google-play-to-prd` 产出的逆向驱动输入文件

如果输入不足，至少要明确：
- 产品目标
- 核心用户
- 核心场景
- 本期范围

如果输入来自逆向流程，优先整理这些文件：
- `product-positioning.md`
- `feature-spec-input.md`
- `page-spec-input.md`
- `api-contracts.md`
- `monetization-impl.md`
- `permissions-compliance.md`
- `technical-architecture.md`
- `code-reuse-notes.md`
- `reverse-evidence-index.md`

---

## 3. 目标
产出一份 **可评审、可设计、可开发、可测试、可埋点** 的 PRD，而不是一个简陋提纲。

---

## 4. 标准输出结构
至少覆盖：
- One Page Summary / 结论摘要
- 文档信息（版本号、作者、日期、审批人、状态）
- 变更记录（Changelog）
- 产品背景 / 问题定义
- 逆向输入摘要（若来自逆向流程）
- 项目目标与非目标
- 成功指标（Success Metrics）
- 目标用户分层
- 使用场景 / JTBD / 用户故事
- 功能范围定义
- 功能清单总览
- 详细功能规格
- 页面级规格
- 交互流程图
- 信息架构 / 页面结构
- 状态与规则设计
- 数据与埋点设计
- 商业化 / 增长设计
- 非功能需求
- 技术实现约束
- API / 数据契约
- 可复现实现建议
- 实验设计与灰度验证
- 验收标准
- 版本规划 / 上线计划
- 风险清单
- 待确认事项
- 相关文档链接（References）

---

## 5. 详细功能规格要求
每个核心功能尽量包括：
- 目标
- 功能说明
- 输入 / 输出
- 前置条件
- 主流程
- 异常流程
- 业务规则
- 状态定义
- 埋点要求
- 验收标准
- 优先级
- 收益判断
- 方案取舍
- 逆向证据引用（若来自逆向流程）

要求：
- 区分“事实 / 推断 / 建议”
- 开发建议尽量回溯到逆向证据或动态验证结果
- 对不确定内容标注“待验证”，不要伪造确定性

---

## 5.1 成功指标（Success Metrics）要求
每份 PRD 必须包含可量化的成功指标：

```markdown
## 成功指标

### 核心指标（判断项目是否成功）
| 指标 | 基线 | 目标 | 挑战 | 数据来源 |
|------|------|------|------|----------|
| [指标名] | [当前值] | [目标值] | [挑战值] | [埋点/后台] |

### 护栏指标（不能变差）
| 指标 | 当前值 | 底线 |
|------|--------|------|
| [指标名] | [当前值] | [不能低于] |

### 评估时间
- 上线后 [X] 天评估核心指标
- 上线后 [Y] 天评估护栏指标
```

原则：
- 每个目标必须有对应的可量化指标
- 指标必须有基线（当前值）和目标值
- 必须明确数据来源和评估时间
- 护栏指标防止“做好了 A 但破坏了 B”

---

## 5.2 交互流程图要求
核心流程必须附带流程图，使用 Mermaid 格式：

```markdown
## 核心流程：[流程名]

\`\`\`mermaid
flowchart TD
    A[开始] --> B{条件判断}
    B -->|是| C[操作 A]
    B -->|否| D[操作 B]
    C --> E[结束]
    D --> E
\`\`\`
```

要求：
- 每个核心功能至少 1 张流程图
- 覆盖主流程 + 关键异常分支
- 使用 Mermaid 格式（可在飞书/GitHub/Notion 直接渲染）
- 复杂流程可用泳道图（swimlane）区分角色

---

## 5.3 变更记录（Changelog）要求
PRD 头部必须包含变更记录表：

```markdown
## 变更记录

| 版本 | 日期 | 作者 | 变更内容 |
|------|------|------|----------|
| v1.0 | YYYY-MM-DD | [作者] | 初稿 |
| v1.1 | YYYY-MM-DD | [作者] | [变更摘要] |
```

原则：
- 每次修改必须新增一行
- 版本号递增（大改 +1.0，小改 +0.1）
- 变更内容简洁明确，方便评审时快速定位差异

---

## 5.4 相关文档链接（References）要求
PRD 末尾必须包含关联文档：

```markdown
## 相关文档

| 文档 | 链接 | 说明 |
|------|------|------|
| 设计稿 | [Figma 链接] | UI/交互设计 |
| 技术方案 | [链接] | 后端/前端技术方案 |
| 竞品分析 | [链接] | 竞品调研报告 |
| 数据看板 | [链接] | 埋点/指标看板 |
| 会议纪要 | [链接] | 评审会议记录 |
```

原则：
- 所有外部依赖文档必须有链接
- 方便评审人快速跳转查看上下文
- 链接失效时应及时更新

---

## 6. 输出原则
- 不要只写方向，要写可执行规则
- 不要省略 In scope / Out of scope / MVP
- 不要只写主流程，必须覆盖异常
- 不要没有验收标准和埋点
- 不要没有成功指标，每个目标必须可量化
- 不要没有流程图，核心流程必须可视化
- 如果输入来自逆向流程，不要只把逆向报告挂附录，必须把逆向结论融合进正文
- API / 权限 / 商业化 / 技术方案要尽量引用逆向证据来源
- 如果用户说“直接生成”，应直接给完整草稿


---

## 7. 联动技能
- 本 skill 是 `google-play-to-prd` 流水线的下游，接收逆向分析结果作为输入
- 多竞品对比时，接收 `competitive-analysis` + `gap-analysis` 的输出
- 效果评估时，结合 `product-data-analysis`
- 原型设计时，结合 `prototype-designer`
