# 1. 目的

本项目拟打造一款 Android 端“可信型删除恢复助手”，满足用户在误删照片、视频、文档后，快速扫描、识别、预览、恢复和整理的需求。

相较于市面上以强情绪文案、强广告承接、强通知召回为主的恢复类工具，本产品强调：
- 结果可解释
- 权限透明
- 恢复来源可标注
- 恢复与整理闭环
- 适度商业化
- 低打扰体验

本文档核心目的：
1. 明确产品目标与非目标
2. 明确首期范围与边界
3. 明确页面、状态、规则、异常逻辑
4. 明确广告、PUSH、埋点、验收和版本规划
5. 为设计、研发、测试提供统一基准

# 5. 需求背景

## 5.1 市场背景
删除恢复与手机清理是长期存在的工具类需求，具备以下特点：
- 用户问题明确，下载动机强
- Android 端权限与文件访问能力更丰富，产品可实现空间更大
- 恢复需求低频但情绪强，适合做强转化入口
- 清理/整理需求高频，可作为复访和留存场景
- 商业化成熟，广告与增值功能空间明确

## 5.2 竞品研究结论
基于对 `com.xhiaguim.isophoto` 的真实安装包逆向，可以确认：
- 竞品并非“专业底层恢复工具”，更像媒体库扫描与结果包装产品
- 它通过恢复叙事获取流量，再用清理、通知和广告体系提升收益
- 其恢复结果高概率混合了：
  - 现存媒体索引
  - 缩略图
  - 缓存资源
  - 可重新索引的旧媒体
- 它不是以“可信解释”取胜，而是更偏“高刺激转化”

## 5.3 业务机会
对我方而言，机会在于做一个与竞品显著区隔的版本：
1. 用“恢复”拿到下载动机
2. 用“结果来源解释”建立用户信任
3. 用“恢复后整理”形成闭环
4. 用更克制的商业化策略控制口碑风险

## 5.4 当前问题定义
行业常见问题：
- 用户误以为扫描结果都是真删除恢复
- 恢复能力被夸大，期望管理失衡
- 广告承接过重，影响核心体验
- 通知频繁，破坏口碑
- 恢复、整理、结果管理没有形成清晰闭环

## 5.5 本项目要解决的问题
- 用户在误删后能迅速进入恢复流程
- 用户能看懂结果“来自哪里”
- 用户能完成筛选、预览、恢复、保存
- 用户恢复后可进一步做清理和整理
- 产品能在不明显破坏体验的前提下实现商业化

# 6. 项目目标与非目标

## 6.1 项目目标
### 产品目标
- 在 Android 端交付一款可用、可信、可解释的恢复工具 MVP
- 建立“恢复 + 整理”双场景产品结构
- 用更透明的结果说明与更低打扰的商业化体验建立差异化

### 业务目标
- 扫描启动率 ≥ 65%
- 扫描完成率 ≥ 75%
- 恢复点击率 ≥ 30%
- 恢复成功保存率 ≥ 20%
- 7日留存 ≥ 12%
- 通知开启率 ≥ 20%
- 广告触发后退出率 < 18%

### 体验目标
- 核心恢复路径不出现无解释的强制广告打断
- 权限申请具备明确说明与二次引导
- 扫描结果默认按价值分层展示

## 6.2 非目标
本期不追求：
- 专业司法取证级恢复
- Root 后底层块扫描恢复
- 全品类文件深度恢复引擎
- 多端同步
- 云恢复/云备份
- 复杂会员订阅体系

# 7. 目标用户

## 7.1 用户分层
### P0 核心用户
- 误删照片 / 视频后有强烈找回需求的普通 Android 用户
- 对系统文件结构不了解，希望一键完成扫描与恢复

### P1 次核心用户
- 手机存储混乱、希望恢复和整理一起完成的用户
- 常拍照、常截图、常下载文件的人群

### P2 可拓展用户
- 对手机文件安全敏感的人群
- 有备份习惯，希望看到更明确结果来源的用户

## 7.2 用户特征
- 情绪强，尤其在误删后急于求解
- 容忍少量商业化，但不接受过度打断
- 对权限敏感，但在高价值场景下可被解释说服
- 对“到底找回了什么”非常在意

## 7.3 核心痛点
- 不知道误删内容是否还能找回
- 不知道哪些结果是真正有用的
- 不知道恢复后保存到了哪里
- 恢复后文件仍然杂乱，需要进一步整理

# 8. 核心场景

## 8.1 场景 1，照片误删恢复
用户删除了相册照片，希望尽快找回。

## 8.2 场景 2，视频恢复
用户误删短视频或录屏，希望定位大文件并恢复。

## 8.3 场景 3，文档恢复
用户误删下载文档、聊天导出文件或工作资料，希望恢复常见文档类型。

## 8.4 场景 4，恢复后整理
用户恢复结束后，希望顺手做去重、截图清理、低价值文件整理。

# 10. 产品流程图（文字版）

## 10.1 首次使用主流程
首页 → 选择恢复类型 → 权限说明页 → 授权成功 → 扫描中页 → 结果页 → 预览/勾选 → 恢复执行 → 恢复完成页 → 已恢复管理 / 去整理

## 10.2 权限拒绝流程
首页 → 恢复类型 → 权限说明 → 用户拒绝 → 说明弹窗 → 再次引导 / 返回首页

## 10.3 恢复后整理流程
恢复完成页 → 推荐整理 → 相似图 / 截图页 → 勾选 → 删除确认 → 完成页

# 11. 功能详细说明

# 11.1 首页

## 11.1.1 目标
让用户在 3 秒内明确可做什么，并快速进入核心恢复流程。

## 11.1.2 页面模块
- 顶部价值说明
- 恢复入口卡片
  - 恢复照片
  - 恢复视频
  - 恢复文件
- 整理空间入口
- 已恢复内容入口
- 底部 Banner 广告位（可配置）

## 11.1.3 业务规则
- 首屏必须突出恢复入口，整理入口作为次级能力
- 首次打开不得在用户未完成核心引导前弹强制插屏
- 若本地存在历史恢复记录，展示“继续查看已恢复内容”快捷入口

## 11.1.4 异常逻辑
- 若存储空间不足，首页展示轻提示，不阻断进入扫描
- 若权限已永久拒绝，点击恢复入口先进入权限说明页而不是直接报错

## 11.1.5 验收标准
- 首页 3 个核心恢复入口可见
- 点击任一入口均进入对应说明或扫描流程
- Banner 加载失败不影响页面操作

---

# 11.2 权限说明页

## 11.2.1 目标
在申请权限前解释：为什么需要权限、能做什么、不能做什么。

## 11.2.2 页面内容
- 权限用途说明
- 恢复边界说明
- 隐私与本地处理说明
- 主按钮：继续授权
- 次按钮：暂不授权

## 11.2.3 业务规则
- 必须明确说明：结果可能包含已有文件、缓存、缩略图等来源
- 不允许使用夸张成功率文案，如“90% 恢复成功”
- 若系统权限弹窗被拒绝一次，页面需给出二次解释
- 若永久拒绝，提供去系统设置页按钮

## 11.2.4 验收标准
- 第一次进入恢复流程必须出现权限说明页
- 权限被拒绝后可正常返回首页
- 永久拒绝时有明确设置引导

---

# 11.3 扫描中页

## 11.3.1 目标
让用户感知扫描正在进行，并对过程有稳定预期。

## 11.3.2 页面元素
- 扫描进度
- 扫描类型说明
- 已发现数量
- 温馨提示：结果将按来源分类展示
- 可取消按钮

## 11.3.3 业务规则
- 扫描过程至少展示以下状态：初始化、扫描中、整理结果、完成
- 每 500ms~1000ms 刷新一次进度或结果数量
- 若可用结果为 0，也必须进入结果页展示空态，而不是只弹 toast

## 11.3.4 异常逻辑
- 权限失效，停止扫描并跳转权限说明页
- 扫描超时，给出失败说明和重试入口
- 设备空间不足，仅提示，允许继续浏览结果

## 11.3.5 验收标准
- 扫描开始后 1 秒内出现动态反馈
- 扫描完成必进入结果页或失败页
- 取消扫描可安全返回首页

---

# 11.4 结果页

## 11.4.1 目标
将扫描结果结构化呈现，让用户理解“哪些更值得恢复”。

## 11.4.2 Tab 分类
- 高价值候选
- 已有文件
- 缓存文件
- 缩略图
- 其他文件

## 11.4.3 默认排序规则
- 默认进入“高价值候选”Tab
- 默认按时间倒序
- 同优先级下按文件大小降序

## 11.4.4 列表项字段
- 缩略图/图标
- 文件名
- 文件类型
- 文件大小
- 时间
- 来源标签
- 勾选状态

## 11.4.5 业务规则
- 必须展示来源标签
- 若为缓存/缩略图类内容，需在 UI 中做弱化提示
- 默认不全选，避免误恢复
- 支持批量勾选与单项预览

## 11.4.6 空态规则
- 各 Tab 均可能空
- 空态需解释“为什么没有结果”或“为什么结果较少”
- 空态可推荐切换到其他恢复类型或进入整理功能

## 11.4.7 广告规则
- 结果页不在首次进入时立即弹插屏
- 用户完成至少一次滚动或一次操作后，才允许触发配置型广告机会

## 11.4.8 验收标准
- 所有结果均能看到来源标签
- 缓存/缩略图内容有差异化展示
- 至少支持单选、多选、全不选、进入预览

---

# 11.5 预览页

## 11.5.1 目标
帮助用户在恢复前确认内容有效性。

## 11.5.2 支持内容
- 图片预览
- 视频封面与基础播放
- 文档类型显示图标、名称和基础信息

## 11.5.3 业务规则
- 预览页允许加入/取消勾选
- 从预览返回后，保留原列表滚动位置与勾选状态
- 若内容无法预览，需明确提示原因，不得直接白屏

## 11.5.4 验收标准
- 预览后返回不丢状态
- 无法预览内容有可读提示

---

# 11.6 恢复执行

## 11.6.1 目标
将用户选择的内容保存到指定恢复目录并给出清晰结果。

## 11.6.2 输入
- 用户选中的文件集合
- 恢复目标路径

## 11.6.3 输出
- 成功数量
- 失败数量
- 失败原因
- 实际保存位置

## 11.6.4 业务规则
- 默认保存到应用恢复目录
- 若同名文件已存在，支持：跳过 / 覆盖 / 自动重命名
- 恢复过程中展示进度
- 恢复结束必须输出成功/失败汇总

## 11.6.5 异常逻辑
- 路径不可写
- 目标空间不足
- 文件源失效
- 部分文件恢复失败

## 11.6.6 验收标准
- 恢复完成后有明确结果页
- 失败项可看到原因分类
- 恢复成功项可直接打开目录或继续整理

---

# 11.7 恢复完成页

## 11.7.1 页面内容
- 恢复成功数
- 恢复失败数
- 保存路径
- 按钮：查看已恢复内容
- 按钮：去整理空间
- 按钮：返回首页

## 11.7.2 业务规则
- 若全部失败，主按钮改为“重新尝试”
- 若部分成功，优先展示成功结果，同时保留失败说明入口
- 恢复完成页允许配置插屏广告，但默认仅在非首次核心流程后触发

## 11.7.3 验收标准
- 所有恢复任务结束后必进入完成页
- 按钮跳转正确

---

# 11.8 已恢复内容管理

## 11.8.1 目标
帮助用户复查和管理已恢复内容，增强结果确定感。

## 11.8.2 功能
- 按类型筛选
- 按时间排序
- 打开文件
- 分享文件
- 删除记录

## 11.8.3 业务规则
- 已恢复记录与真实文件状态保持尽可能一致
- 文件缺失时需显示“文件已不存在”

# 11.9 整理空间模块

## 11.9.1 首期范围
- 相似图整理
- 截图整理

## 11.9.2 目标
承接恢复后的次需求，提高复访与停留，同时真正创造使用价值。

## 11.9.3 业务规则
- 与恢复结果页面结构保持一致
- 删除前必须二次确认
- 删除后需有可回看结果页

# 12. 广告相关

## 12.1 设计原则
- 广告不能破坏首次核心恢复体验
- 广告应优先出现在阶段性完成节点，而不是关键决策节点
- 激励广告应替代一部分强制插屏
- 所有广告频控、冷却、平台切换需配置化

## 12.2 首期广告点位表

| 序号 | 广告位 | 页面/场景 | 类型 | 默认策略 | 备注 |
|---|---|---|---|---|---|
| 1 | openPageAdv | 冷启动首页 | 开屏 | 可配，首次默认关闭 | 首次使用不打断核心引导 |
| 2 | homeBottomAdv | 首页底部 | Banner | 开启 | 不影响主操作 |
| 3 | resultSoftInterstitial | 结果页发生一次有效操作后 | 插屏 | 低频 | 仅非首次用户默认可开 |
| 4 | recoverFinishAdv | 恢复完成页后 | 插屏 | 中频 | 可配置概率 |
| 5 | cleanBottomAdv | 整理页底部 | Banner | 开启 | 稳定承接 |
| 6 | cleanFinishAdv | 整理完成页 | 插屏 | 中频 | 收尾型 |
| 7 | extraRecoverQuotaReward | 恢复更多数量时 | 激励 | 开启 | 用户主动触发 |

## 12.3 广告频控规则
- 参数 `n`：插屏触发频率，默认 3
- 参数 `max`：24 小时内最大插屏展示次数，默认 8
- 参数 `cooldownHours`：达到上限后的冷却时长，默认 24 小时
- Banner 不参与插屏上限，但应支持全局关闭
- 激励广告不计入强制插屏上限

## 12.4 广告异常处理
- 开屏广告加载超时上限 3 秒
- 插屏广告加载超时上限 2 秒
- 超时或失败直接放行主流程
- 所有广告失败不得阻塞恢复功能

## 12.5 配置项
- 平台优先级
- 点位开关
- 展示概率
- 频控参数
- 首次用户豁免开关
- 归因维度广告策略开关

# 13. PUSH / 通知相关

## 13.1 目标
在不明显骚扰用户的前提下，提高回流率和恢复完成率。

## 13.2 推送类型
| 类型 | 说明 | 是否首期 |
|---|---|---|
| 系统权限引导通知 | 用户中断流程后的轻提醒 | 是 |
| 恢复完成提醒 | 恢复任务完成后的结果提醒 | 是 |
| 定期扫描召回 | 推荐用户再次清理/检查 | 否，二期 |
| 可恢复结果召回 | 有历史扫描记录时二次提醒 | 否，二期 |

## 13.3 跳转规则
| 推送类型 | 点击后跳转 |
|---|---|
| 权限引导通知 | 权限说明页 |
| 恢复完成提醒 | 已恢复内容页 |
| 定期扫描召回 | 首页 |
| 可恢复结果召回 | 对应结果页或首页 |

## 13.4 频控规则
- 首期默认不启用高频营销式 PUSH
- 每日最多 1 条主动召回类通知
- 7 日内主动召回上限 3 条
- 用户关闭通知后不再二次强引导

## 13.5 文案原则
- 不使用“最后机会”“时间不多了”“90% 成功率”式夸张文案
- 使用中性、可信、可解释表达
- 强调“查看结果”而不是“立即抢救”

# 14. 埋点设计

## 14.1 埋点原则
- 覆盖从首页进入、授权、扫描、浏览、恢复、整理、广告、通知的完整链路
- 所有关键步骤必须有事件与参数
- 能区分来源类型、恢复类型、广告触发结果、通知开关状态

## 14.2 核心事件表

| 事件名 | 触发时机 | 核心参数 | 目的 |
|---|---|---|---|
| home_view | 首页曝光 | is_first_open | 统计首页访问 |
| recover_entry_click | 点击恢复入口 | entry_type | 统计入口偏好 |
| permission_explain_view | 权限说明页曝光 | recover_type | 统计授权前流失 |
| permission_result | 权限申请结果 | recover_type, result | 统计授权转化 |
| scan_start | 开始扫描 | recover_type | 统计扫描启动率 |
| scan_finish | 扫描完成 | recover_type, result_count, duration | 统计扫描效率 |
| result_tab_view | 查看结果 Tab | tab_type | 统计结果结构点击 |
| preview_open | 打开预览 | file_type, source_tag | 统计高意向行为 |
| recover_click | 点击恢复 | selected_count, recover_type | 统计恢复意图 |
| recover_finish | 恢复完成 | success_count, fail_count | 统计恢复成效 |
| clean_entry_click | 点击整理入口 | clean_type | 统计二次场景承接 |
| ad_request | 请求广告 | ad_slot, ad_type | 广告健康度 |
| ad_show | 广告展示 | ad_slot, ad_type | 广告收入分析 |
| ad_skip_by_fail | 广告失败放行 | ad_slot | 体验保护监控 |
| notification_permission_result | 通知权限结果 | result | 统计召回能力基础 |
| push_click | 点击通知 | push_type | 统计召回效果 |

## 14.3 关键分析维度
- 恢复类型
- 首次/非首次用户
- 来源标签分布
- 文件类型
- 广告触发前后的退出率
- 通知开关状态
- 不同归因渠道用户表现

# 15. 非功能需求

## 15.1 性能
- 首页首屏可交互时间 < 2 秒
- 扫描中页在点击后 1 秒内展示反馈
- 结果页切换 Tab 响应时间 < 500ms

## 15.2 稳定性
- 恢复流程崩溃率 < 0.5%
- 广告失败不影响主流程可用性
- 权限拒绝与恢复失败场景有完整兜底

## 15.3 隐私与合规
- 权限申请需有明确用途说明
- 不得夸大恢复能力
- 通知权限不作为核心功能强绑定前置条件
- 恢复数据优先本地处理

## 15.4 可配置性
- 广告开关、频控、概率需服务端可配
- PUSH 开关与频控需服务端可配
- 首页文案与实验位可配置

# 16. 技术实现约束

## 16.1 事实约束
基于竞品逆向结论与 Android 平台现状，首期实现应默认建立在以下事实之上：
- 恢复主能力更接近 `MediaStore` 与文件索引聚合
- 不承诺专业块扫描恢复
- 结果需做来源分类与可信解释

## 16.2 依赖
- Android 存储访问能力
- 媒体索引扫描能力
- 本地恢复目录写入能力
- 广告 SDK 与配置平台
- PUSH / FCM 基础设施
- 埋点 SDK 与数据后台

## 16.3 假设
- 用户大部分为非 Root Android 用户
- 用户会接受合理的权限说明，只要核心价值足够清晰
- 首期恢复体验的主要价值来自“可发现 + 可解释 + 可管理”，而不是取证级恢复

# 17. 验收标准

## 17.1 产品验收
- 首次用户可完整走通：首页 → 授权 → 扫描 → 结果 → 恢复 → 完成
- 结果页包含来源分类，不是单一混合列表
- 恢复完成页可查看已恢复内容
- 整理模块可从恢复完成页承接进入

## 17.2 研发验收
- 权限拒绝、永久拒绝、空间不足、恢复失败、广告失败均有兜底
- 所有核心事件埋点已接入
- 广告点位和频控参数均可配置
- 通知权限与 Push 开关不阻断主链路

## 17.3 测试验收
至少覆盖以下用例：
- 首次安装首开
- 非首次打开
- 权限一次拒绝
- 权限永久拒绝
- 扫描结果为空
- 扫描结果很多
- 部分恢复失败
- 广告加载失败
- 通知权限关闭
- 恢复后整理链路

# 19. 风险清单

| 风险 | 描述 | 应对 |
|---|---|---|
| 恢复能力被误解 | 用户期待“真正底层恢复” | 强化边界说明与结果标签 |
| 广告过重导致差评 | 关键路径广告过强 | 首次链路减打扰，频控可配 |
| 权限敏感导致流失 | 用户拒绝存储权限 | 强化解释页和二次引导 |
| 结果质量不稳定 | 不同设备文件索引差异大 | 做来源分类和空态解释 |
| 通知被视为骚扰 | 召回频率过高 | 首期克制，默认低频 |

# 23. 状态机设计

## 23.1 扫描流程状态机
| 当前状态 | 触发条件 | 下一状态 | 用户可见表现 |
|---|---|---|---|
| idle | 用户点击恢复入口 | permission_check | 进入权限说明或直接校验权限 |
| permission_check | 权限已授权 | scanning | 展示扫描中页 |
| permission_check | 权限未授权 | permission_explain | 展示权限说明页 |
| permission_explain | 用户授权成功 | scanning | 开始扫描 |
| permission_explain | 用户拒绝 | permission_denied | 展示拒绝说明 |
| scanning | 扫描完成且有结果 | result_ready | 进入结果页 |
| scanning | 扫描完成且无结果 | result_empty | 进入空态结果页 |
| scanning | 扫描失败 | scan_failed | 展示失败说明与重试 |
| result_ready | 用户点击恢复 | recovering | 进入恢复执行态 |
| recovering | 全部成功 | recover_success | 进入完成页 |
| recovering | 部分失败 | recover_partial_success | 展示成功失败汇总 |
| recovering | 全部失败 | recover_failed | 展示失败结果页 |

## 23.2 权限状态机
| 状态 | 条件 | 页面/动作 |
|---|---|---|
| unknown | 首次进入 | 展示权限说明页 |
| granted | 用户同意 | 开始扫描 |
| denied_once | 用户拒绝一次 | 展示二次解释 |
| denied_forever | 用户永久拒绝 | 引导去设置页 |

## 23.3 广告状态机
| 状态 | 触发 | 下一状态 | 说明 |
|---|---|---|---|
| ad_idle | 进入广告机会点 | ad_loading | 开始请求广告 |
| ad_loading | 广告加载成功 | ad_showing | 展示广告 |
| ad_loading | 广告加载失败/超时 | ad_skip | 直接放行 |
| ad_showing | 广告关闭 | ad_finish | 返回业务流程 |
