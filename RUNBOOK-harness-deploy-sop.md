# Harness / OpenClaw 能力投放 SOP

适用场景：
- Python → Node.js 迁移
- 新模块接入执行链
- 补齐缺失核心能力
- 做“可投放版本”而不是纯开发态代码

---

## 一、目标定义

投放目标不是“代码写完”，而是达到这 4 个标准：

1. **能进入主执行链**
2. **能完成最小闭环**
3. **能处理常见输入**
4. **出错时不会把系统拖垮**

一句话：

**可投放 = 可调用 + 可执行 + 可返回 + 可兜底。**

---

## 二、标准工作流

### 阶段 0：接单与归类

先判断本次需求属于哪一类：

- **修复型**：已有链路异常，补漏洞
- **迁移型**：从旧实现迁到新实现
- **新增型**：加入新能力
- **验收型**：检查是否达到上线条件

#### 输出物
- 一句话任务定义
- 本次范围（in scope）
- 不做的部分（out of scope）

#### 模板
- **任务**：补齐 Node 版工具执行链缺失能力
- **范围**：tools / middlewares / context / agent loop
- **暂不处理**：低频增强功能、UI、美化类封装

---

### 阶段 1：差距分析（Gap Analysis）

核心问题只有三个：

1. **现在已有啥**
2. **还缺啥**
3. **哪个缺口最影响投放**

#### 必查项
- 目录结构是否齐
- 核心文件是否存在
- 关键入口是否能串起来
- 是否有替代实现但未接入
- 是否有“看起来有，实际上不能用”的假覆盖

#### 输出物
- 缺失模块清单
- 每个模块影响级别：P0 / P1 / P2
- 本轮优先修复顺序

#### 优先级定义
- **P0**：不补就跑不通
- **P1**：能跑但高频场景会炸
- **P2**：优化体验，不影响上线

---

### 阶段 2：锁定最小可运行闭环

不要一上来追求全量覆盖。
先定义本次投放最小闭环（MRC: Minimum Runnable Closure）。

#### 一个能力可投放，至少要有：
- 入口
- 调度
- 执行
- 返回
- 错误处理

#### 对 Harness / OpenClaw 常见映射
- `agent-loop`
- `context-lifecycle`
- `tools-executor`
- `middlewares-executor`

#### 判断标准
如果这条链路中任意一点缺失，就不能叫“可投放”。

---

### 阶段 3：按优先级补齐

#### 3.1 先补 P0 主链路

优先处理：
- 执行入口
- 调度逻辑
- 工具执行
- 中间件执行
- 上下文生命周期

##### 原则
- 先打通，不先做漂亮
- 先保证可运行，不先做全配置化
- 先补真实缺口，不补表面覆盖率

---

#### 3.2 再补 P1 高频保障能力

这些通常决定“上线后会不会频繁炸”：

- 大结果持久化
- 智能截断输出
- 参数自动修正
- skills 文件读取
- 子代理派发
- 关键控制型工具（如 stop_dev_server）

##### 原则
- 只补高频、刚需、明显增稳的能力
- 一切以降低失败率和上下文污染为先

---

#### 3.3 最后看 P2 增强项

例如：
- 更细的日志分类
- 更完整 telemetry
- 更优雅错误文案
- 附加兼容层

这些可以延后，不阻塞投放。

---

## 四、实现原则

### 原则 1：投放优先，不做假完整
避免这种情况：
- 文件存在，但没接入口
- 函数存在，但调用不到
- 能处理 happy path，但边界全炸

结论：
**代码存在 ≠ 能投放。**

---

### 原则 2：优先真实调用链
实现顺序应按真实调用顺序走，不按文件名顺序走。

先看：
1. 谁是入口
2. 入口依赖谁
3. 谁是执行器
4. 执行结果怎么回收

---

### 原则 3：高风险点必须先兜底
对这类点要优先加保护：

- 超长输出
- 缺参
- 参数格式错误
- 工具报错无分类
- 子任务结果失控
- 大对象写回上下文

---

### 原则 4：兼容旧行为，但不盲目复刻
迁移不是逐字符复制。
应保留的是：
- 行为语义
- 主链路能力
- 关键边界处理

不必机械照搬：
- 旧命名
- 旧内部结构
- 旧冗余逻辑

---

## 五、验收 SOP

投放前验收至少做三层。

### 第一层：结构验收
检查：
- 文件存在
- 导出正确
- import / require 路径正确
- 无明显缺依赖
- 无明显语法错误

#### 通过标准
“系统至少能加载，不在启动阶段直接爆炸”

---

### 第二层：链路验收
检查：
- 能进入入口
- 能走到调度
- 能执行工具
- 能经过中间件
- 能返回结果
- 错误时能收口

#### 通过标准
“主链路通，失败也可控”

---

### 第三层：场景验收
至少跑 3 类样例：

1. **正常短任务**
2. **大输出任务**
3. **边界/异常任务**

#### 关注点
- 是否上下文爆炸
- 是否错误信息可读
- 是否输出被合理截断/持久化
- 是否出现隐性死链

#### 通过标准
“真实任务可跑，非 happy path 也不致命”

---

## 六、投放判定标准

满足以下条件即可判定“可投放”：

- 核心链路齐全
- 高优先级缺口已补
- 主流程可运行
- 常见异常有兜底
- 不会明显污染上下文
- 不会因单次任务导致系统不稳定

### 不要求
- 覆盖所有增强功能
- 完美日志体验
- 全量历史能力完全一致
- 所有低频工具一次性补齐

---

## 七、投放后观察 SOP

投放后不要立刻进入“完成幻觉”，要观察这几类问题：

### 观察项
- 新报错类型是否出现
- 自动修正规则是否还不够
- 是否仍有超长输出压垮上下文
- 是否有模块“存在但没被真实调用”
- 子代理/工具调用是否有兼容断层

### 常见后续补强项
- 增加 auto-fix 规则
- 优化错误分类
- 完善大结果落盘策略
- 补低频但关键的边界场景

---

## 八、文档输出模板

每次投放完成后，建议固定输出一份简报：

### 1. 本次目标
一句话说明本次修什么。

### 2. 差距结论
列出发现的核心缺失。

### 3. 本次完成
列出补齐的文件/模块/能力。

### 4. 投放判断
说明为什么现在可以投放。

### 5. 未纳入部分
列出本轮没做但后续可补的内容。

---

## 九、适合当前工程的推荐版本

### 当前推荐投放顺序

#### Step 1：先做差距分析
确认：
- Python 原实现有哪些关键能力
- Node 当前少了哪些 P0/P1 模块

#### Step 2：锁定核心 4 件套
优先确保：
- `agent-loop`
- `context-lifecycle`
- `tools-executor`
- `middlewares-executor`

#### Step 3：补高频稳定器
优先补：
- `read_skill_file`
- `smartTruncateOutput`
- `persistLargeResult`
- `delegate_task`
- `stop_dev_server`
- `autoFixArgs`

#### Step 4：做最小验收
跑：
- 正常任务
- 长输出任务
- 异常参数任务

#### Step 5：确认可投放
只要主链路通、异常可控，就可以先投放。

## 十、极简版 Checklist

以后自己过一眼这个就够了：

- [ ] 任务已归类（修复 / 迁移 / 新增 / 验收）
- [ ] 已完成 gap analysis
- [ ] 已识别 P0 / P1 / P2
- [ ] P0 主链路已补齐
- [ ] P1 高频稳定能力已补关键项
- [ ] 入口可达
- [ ] 执行可达
- [ ] 返回可达
- [ ] 错误可收口
- [ ] 大输出可控
- [ ] 最小场景验收已跑
- [ ] 已确认“可投放”而非“表面完成”

---

## 十一、一句话方法论

**先补能跑的骨架，再补高频防炸件，最后再做增强。**
