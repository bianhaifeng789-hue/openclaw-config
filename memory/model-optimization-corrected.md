# 模型调用优化方案（修正版）

## 用户反馈

**关键洞察**: 模型降级会导致 AI"降智"，这不是好方案。

**正确方向**: 保持智能，优化执行方式，而非降低能力。

---

## 重新评估优化方向

### ❌ 不推荐的方案

| 方案 | 问题 | 为何不好 |
|------|------|---------|
| **模型降级** | 快但不够智能 | glm-4-flash 能力低于 glm-5，复杂任务会出错 |
| **Temperature降低** | 降低创造力 | 快但回答僵硬 |
| **Thinking禁用** | 降低推理质量 | 快但深度分析会变差 |

### ✅ 推荐的方案

| 方案 | 优势 | 为何好 |
|------|------|--------|
| **spawn 后台执行** | 保持智能 + 不阻塞 | 主模型能力不变，长任务并行 |
| **上下文压缩** | 保持智能 + 更快 | 减少 context tokens，推理更快 |
| **工具缓存** | 保持智能 + 减少调用 | 相同查询即时返回，不重复调用 |
| **Thinking 按需启用** | 保持智能 + 按需快 | 简单任务快，复杂任务深度思考 |

---

## 核心洞察

**问题本质**: 不是模型慢，而是执行方式串行阻塞。

**正确优化**: 改执行方式，不改模型能力。

---

## 推荐方案详解

### 方案1: spawn 后台执行（核心推荐）

**原理**: 长任务不影响主线程

**效果对比**:
```
现在:
用户: 移植 DeerFlow
我: [执行40分钟，阻塞，你无法聊天]
你: [焦虑等待，不知道进度]

优化后:
用户: 移植 DeerFlow
我: 收到！开始后台执行（预估10分钟）
    你可以继续和我聊天，我会报告进度。
    [spawn subagent 执行]
    
用户: 进度如何
我: 60% (3/5)，正在创建 Skills

用户: 顺便帮我升级 OpenClaw
我: 好的，并行执行中...
    [spawn 另一个 subagent]

我: ✅ DeerFlow 移植完成
我: ✅ OpenClaw 升级完成
```

**优势**:
- ✅ 保持 glm-5 智能不变
- ✅ 主线程即时响应
- ✅ 长任务并行执行
- ✅ 用户体验可控

### 方案2: 上下文管理优化

**原理**: 减少 context tokens，推理更快

**具体措施**:

#### 2.1 自动压缩阈值降低
```
现在: 167k (93%) → 压缩
优化: 80k (44%) → 压缩

效果: context 保持更精简，模型推理更快
```

#### 2.2 工具结果精简
```
现在: 保留所有工具结果（完整内容）
优化: 只保留摘要 + 最近5次完整结果

效果: 减少 50-70% tokens
```

#### 2.3 消息历史摘要
```
现在: 保留所有对话历史
优化: 旧对话自动摘要（保留关键信息）

效果: 100条消息 → 10条摘要
```

### 方案3: 工具调用优化

**原理**: 减少 API 调用次数

#### 3.1 文件读取缓存
```javascript
// 缓存文件内容
const fileCache = new Map();

async function readCached(filePath) {
  if (fileCache.has(filePath)) {
    return fileCache.get(filePath);  // 即时返回
  }
  
  const content = await readFile(filePath);
  fileCache.set(filePath, content);
  return content;
}
```

#### 3.2 工具调用合并
```javascript
// 旧模式：多次调用
await read("file1.txt");
await read("file2.txt");
await read("file3.txt");
// 总延迟：3 * 10s = 30s

// 新模式：一次调用多个
await readBatch(["file1.txt", "file2.txt", "file3.txt"]);
// 总延迟：10s（并行读取）
```

### 方案4: Thinking 模式智能控制

**原理**: 按需启用，而非总是启用

```yaml
thinking:
  mode: "adaptive"  # 自适应模式
  
  rules:
    # 禁用 thinking（快速）
    simple_query: false      # "你好"、"进度如何"
    file_operation: false    # 读文件、写文件
    
    # 启用 thinking（深度）
    code_generation: true    # 生成代码
    deep_analysis: true      # 深度分析
    planning: true           # 任务规划
```

**效果**:
- 简单任务：节省 5-10s（thinking延迟）
- 复杂任务：保持深度推理

## 结论

**错误方向**: 模型降级（快但不够智能）

**正确方向**: 
1. 改执行方式（spawn并行）
2. 减少重复调用（缓存）
3. 按需启用深度思考（Thinking自适应）
4. 保持精简上下文（压缩）

**核心理念**: 保持智能，优化执行。
