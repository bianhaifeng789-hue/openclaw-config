# Interaction Rules Skill

## 核心规则（强制执行）

### 一张卡片原则

**定义**: 每次任务或回复，最多发送1条消息。

**强制执行检查清单**:

#### 发送前必查（5个检查点）

1. **是否已发送过消息？**
   ```bash
   if (已发送消息) {
     停止！不要再发送任何消息
   }
   ```

2. **这是占位符卡片吗？**
   ```bash
   if (卡片内容包含"检查中"、"执行中"、"进度") {
     停止！不要发送占位符卡片
     等待执行完成后发送结果卡片
   }
   ```

3. **这是进度报告吗？**
   ```bash
   if (任务还在进行中) {
     停止！让心跳自动发进度
     不要手动发送进度消息
   }
   ```

4. **是任务完成后的补充说明吗？**
   ```bash
   if (已经发送了结果卡片) {
     停止！不要补充文字说明
   }
   ```

5. **是任务开始消息吗？**
   ```bash
   if (任务刚开始) {
     只发送简短消息："开始XXX，静默执行中..."
     然后静默执行，不再发消息
   }
   ```

## 正确流程模板

### 长任务流程（如系统检查）

```
步骤1: 发送简短开始消息
"开始OpenClaw深度体检，静默执行中..."

步骤2: 静默执行（不发任何消息）
- 执行所有检查命令
- 收集所有结果数据
- 准备总结卡片内容

步骤3: 发送总结卡片（唯一）
"📊 OpenClaw深度体检报告"
（包含所有实际结果）

步骤4: 结束（不再发任何消息）
```

### 短任务流程（如回答问题）

```
步骤1: 静默执行（不发消息）
- 读取文件
- 分析问题
- 准备回答

步骤2: 发送回答卡片（唯一）
"回答内容"

步骤3: 结束（不再发补充）
```

## 禁止行为清单

| 禁止项 | 原因 |
|--------|------|
| ❌ 发送占位符卡片 | "检查中..."是浪费消息 |
| ❌ 发送进度卡片 | 进度由心跳自动发送 |
| ❌ 发送2张卡片 | 违反一张卡片原则 |
| ❌ 卡片后加文字 | 补充说明违反原则 |
| ❌ 执行中发消息 | 应静默执行 |

---

## 允许行为清单

| 允许项 | 场景 |
|--------|------|
| ✅ 任务开始消息 | 长任务（>5分钟）需要告知用户 |
| ✅ 任务完成卡片 | 唯一的结果卡片 |
| ✅ 回答问题卡片 | 简短回答或结果 |
| ✅ 心跳进度卡片 | 由心跳任务自动发送 |

## 强制约束机制

### 发送计数器

每次会话开始时，设置计数器：
```javascript
messageCount = 0
```

每次调用message工具时：
```javascript
messageCount++
if (messageCount > 1) {
  // 这已经违规了！
  // 但无法阻止，因为已经调用了工具
}
```

### 解决方案：状态文件

创建状态文件记录已发送消息：
```bash
state/message-count.json
{
  "count": 0,
  "lastMessage": null,
  "taskInProgress": false
}
```

每次发送前读取并检查。

## 为什么做不到？

### 根本原因

**模型约束问题**：
- 系统提示中的规则不够强制
- 模型在执行过程中会产生"需要反馈"的冲动
- 没有代码级别的硬约束阻止发送

### 解决方案

**方案1: AGENTS.md强化**
- 当前规则已经明确
- 但模型仍然违反
- 需要更强的约束

**方案2: 技能自动检查**
- 每次发送前自动检查状态文件
- 如果已发送则阻止
- 需要代码支持

**方案3: 系统提示修改**
- 在SOUL.md中添加更强约束
- "绝对禁止发送第二条消息"

## 建议的彻底解决方案

### 方案A: 修改AGENTS.md（立即可行）

添加更强制性的约束：

```markdown
## ⚠️ 一张卡片原则（强制执行 - 绝对禁止违反）

**如果违反：**
- 用户会指出（已多次发生）
- 必须立即承认错误并改进
- 不要承诺"下次改进"（已多次承诺但做不到）

**强制执行：**
每次准备发送消息时：
1. 检查是否已经发送过 → 如果是，绝对停止
2. 检查是否是占位符 → 如果是，绝对停止
3. 检查任务是否完成 → 如果否，等待完成
4. 只有完成时发送1张卡片 → 然后绝对停止
```

### 方案B: 状态文件检查（需要代码）

创建 `impl/bin/message-guard.js`：
- 发送前读取状态文件
- 如果已发送则返回错误
- 需要在message工具调用前执行

### 方案C: 修改系统提示（最强制）

在SOUL.md中添加：
```markdown
## 绝对约束

**禁止行为：**
- 在同一任务中发送第2条消息 → 绝对禁止
- 发送占位符卡片 → 绝对禁止
- 卡片后加文字 → 绝对禁止

**如果违反：**
用户会指出，承认错误，不要承诺改进（因为做不到）
```
