# 交互优化建议

## 观察到的模式

### 问题1: 消息重复发送
**现象**: 同一条回复会重复发送 3-4 次
**原因**: 可能是工具调用失败后的重试机制
**影响**: 用户困惑，信息冗余

**建议**:
- 工具失败时静默重试，不发送失败提示给用户
- 只在最终成功或完全失败时通知用户
- 限制重试次数（最多 2 次）

### 问题2: 工具失败提示过多
**现象**: `⚠️ 📝 Edit: in ... failed` 频繁出现
**原因**: edit 工具匹配失败，但操作后续成功
**影响**: 干扰用户阅读，产生不必要的焦虑

**建议**:
- 使用 write 替代 edit（更可靠）
- edit 失败时自动切换到 write
- 不输出失败提示，静默处理

### 问题3: 进度报告过于频繁
**现象**: 每个小步骤都发送飞书卡片
**原因**: 习惯性地报告每个操作
**影响**: 信息过载，重要信息淹没

**建议**:
- 只在重要里程碑发送卡片（完成、错误、需要确认）
- 进度合并报告（如 "已完成 3/5 步骤"）
- 使用状态文件而非频繁通知

### 问题4: 心跳响应不够静默
**现象**: `HEARTBEAT_OK` 有时附带说明
**原因**: 想解释为什么是 OK
**影响**: 违反静默回复规则

**建议**:
- 严格执行 `HEARTBEAT_OK` = 仅此一行
- 无需解释正常状态
- 仅在异常时发送通知

## 具体改进措施

### 1. 工具调用策略
- edit → 失败时静默切换 write
- 多个相关操作 → 合并在一个工具调用块
- 长任务 → spawn subagent，避免阻塞

### 2. 通知策略
| 场景 | 通知 | 频率 |
|------|------|------|
| 开始任务 | ❌ 不通知 | - |
| 进度更新 | ❌ 不通知 | - |
| 工具失败 | ❌ 不通知 | - |
| 任务完成 | ✅ 飞书卡片 | 仅一次 |
| 需要确认 | ✅ 飞书卡片 | 仅一次 |
| 心跳正常 | ❌ HEARTBEAT_OK | 仅此一行 |

### 3. 错误处理策略
- 工具失败 → 静默重试（最多 2 次）
- 重试失败 → 切换替代方案
- 无法完成 → 发送错误卡片（仅一次）

### 4. 进度追踪策略
- 使用状态文件而非频繁报告
- 飞书卡片仅用于里程碑
- 内部日志记录详细进度

## 实施建议

### 立即可改（无需配置）
1. 减少 `edit` 使用，优先 `write`
2. 合并相关工具调用
3. 仅在完成/错误时发送卡片
4. `HEARTBEAT_OK` 严格单行

### 需要配置
1. 工具重试策略（最多 2 次）
2. 飞书卡片发送阈值
3. 进度报告频率限制
