# Karpathy Coding Guidelines

当任务涉及 **编码、改配置、改脚本、排障修复、实现功能** 时，可用本 skill 作为轻量行为约束。

## 核心四条

### 1. Think Before Coding
先想清楚要解决的到底是什么问题，再动手。

要求：
- 先识别真实目标，不为模糊想法乱铺方案
- 如果用户目标已经明确，就直接走最小实现路径
- 如果存在关键歧义，只问影响实现的那一个，不展开长分析

### 2. Simplicity First
默认优先最简单、最直接、最少依赖的方案。

要求：
- 不为了一次小需求引入新抽象层
- 不为“更优雅”而扩大改动面
- 能用一处小改解决，就不要做系统级重构

### 3. Surgical Changes
只改和当前任务直接相关的地方。

要求：
- 不顺手清理无关文件
- 不顺手修一串旁支问题
- 不扩展任务边界，除非当前任务被它们真实阻塞

### 4. Verify Every Step
每一步都要能验证，不靠感觉说“应该好了”。

要求：
- 代码改完要有最小验证动作
- 配置改完要确认生效位置和结果
- 脚本改完要跑一次最小可观察检查
- 输出结论时区分“已验证”和“推测”

---

## OpenClaw 本地化解释

在当前 OpenClaw 环境中，这四条落地为：

- **先想清楚，但别空转**
- **先做最小改动，不要顺手重构**
- **只碰当前任务相关文件**
- **每一步都给验证动作**
- **结论比铺垫重要**

---

## 默认执行顺序

处理工程任务时，默认顺序：
1. 找到最小改动点
2. 直接修改
3. 做最小验证
4. 报告关键结果和残留风险

---

## 适用场景

- 修一个 bug
- 加一个小功能
- 调整配置
- 增加一个脚本
- 做低风险重构
- 本地集成新能力

---

## 不适合的场景

- 需要大范围重新设计架构
- 用户明确要求做全面规划/长文方案
- 多系统联动且必须先做全局设计

---

## 输出要求

采用这个 skill 时，输出尽量符合：
- 简短
- 直接
- 以动作和验证为中心
- 少铺垫
- 不夸大完成度

---

## 一句话版本

> 先想清楚，先做最小改动，只改必要部分，每一步都验证。
