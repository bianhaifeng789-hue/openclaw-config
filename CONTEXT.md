# CONTEXT.md - 上下文压缩与会话治理规则

目标：减少主会话膨胀、降低 context overflow 概率、把长期信息尽量外化到文件而不是聊天线程。

## 核心原则

- 一个线程尽量只做一类事情：诊断、修复、验证、复盘不要长期混在同一主线程里。
- 长日志、长工具输出、原始诊断细节优先落文件；对话里优先保留摘要、结论、下一步动作。
- 重大异常或排障完成后，优先更新：
  - `memory/YYYY-MM-DD.md`
  - `MEMORY.md`
  - `RUNBOOK.md`
  - `SCRIPTS.md`（如涉及本地自动化）
- 心跳（HEARTBEAT）只做轻检查，不承担大批量监控、排障、总结工作。

## 推荐工作流

1. 先诊断主因
2. 将大段日志/工具输出写入文件或仅保留必要摘要
3. 执行小范围、可验证、可回滚的修复
4. 验证结果
5. 将结论外化到 memory / runbook / scripts / todo
6. 当前线程收口，必要时新开线程处理下一个主题

## 收口触发（默认优先级很高）

满足以下任一条件时，应优先考虑收口当前线程：
- 同一线程里已经在处理多个主题
- 工具输出/日志明显偏长
- 已经完成一轮诊断 + 修复 + 验证
- 已出现 context overflow、compaction 压力或会话明显变胖

默认收口动作：
1. 用 3-6 行写当前结论摘要
2. 更新相关文件（`memory/YYYY-MM-DD.md` / `MEMORY.md` / `RUNBOOK.md` / `SCRIPTS.md` / `OPENCLAW_TODO.md`）
3. 如下一步属于不同主题，优先切到新线程继续

## 避免的做法

- 在同一线程里长期堆叠多类问题（配置、日志、脚本、cron、安全、规划）
- 把大量原始日志直接留在主对话里
- 让主线程承担长期记忆，而不更新文件
- 为了“不断线”而恢复粗暴 watchdog / keeper / cleanup 脚本

## 文件分工

- `memory/YYYY-MM-DD.md`：当天事件与摘要
- `MEMORY.md`：长期规律与应该记住的结论
- `RUNBOOK.md`：排障顺序与操作手册
- `SCRIPTS.md`：本地自动化现状与禁忌
- `OPENCLAW_TODO.md`：未来优化项
- `INCIDENT_TEMPLATE.md`：重大事件模板
- `MEMORY_FLOW.md`：记忆流转规则

## 一句话规则

能写进文件的，就不要一直背在主会话里；能用摘要表达的，就不要把原始长输出反复塞进上下文。
